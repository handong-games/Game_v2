# Combat Design Draft

## 목적

전투는 몬스터를 처치하는 단일 미니게임이 아니라, 플레이어가 현재 스킬 구성의 강점과 약점을 체감하고 다음 보상 선택을 고민하게 만드는 장치다.

전투 중에는 동전 결과와 몬스터 행동 예고를 보고 손실을 줄이는 선택을 한다. 전투 후에는 방금 드러난 약점을 보완할지, 강점을 강화할지, 더 위험한 고점을 노릴지 선택한다.

## 목표 재미

핵심 재미는 운을 이기는 것이 아니라, 운이 만든 상황을 해석하고 수습하는 것이다.

```text
불확실한 동전 결과를 보고,
내가 감당할 수 있는 위험을 선택하며,
그 결과를 내 판단의 책임으로 받아들이는 재미
```

플레이어가 실패했을 때 "게임이 나를 죽였다"가 아니라 "내가 그 위험을 감수했다"에 가깝게 느끼는 것이 중요하다.

## 플레이어 감정 흐름

```text
1. 예고
   몬스터가 다음 행동을 보여준다.
   감정: 이번 턴 위험하네.

2. 불확실성
   동전을 던진다.
   감정: 뭐가 나올까?

3. 해석
   나온 앞면/뒷면 결과로 가능한 스킬이 달라진다.
   감정: 이 결과로 뭘 할 수 있지?

4. 선택
   공격, 방어, 회복, 긴급 대응 중 고른다.
   감정: 이 정도 위험은 감수하자.

5. 결과
   피해를 막거나, 공격하거나, 손해를 본다.
   감정: 내 선택이 이렇게 이어졌구나.

6. 학습
   다음 보상 선택 또는 다음 전투에서 더 나은 선택을 생각한다.
   감정: 다음엔 이 약점을 보완해야겠다.
```

## 전투 정보 공개 원칙

플레이어가 실패를 자신의 선택 결과로 받아들이려면, 중요한 위험 정보가 사전에 보여야 한다.

확정 정보:

```text
1. 몬스터의 다음 행동과 예상 피해
2. 현재 사용할 수 있는 스킬과 필요한 동전 조건
3. 이번 턴에 받을 수 있는 최악의 피해량
```

향후 보조 정보:

```text
4. 안전 선택 / 위험 선택 / 긴급 선택의 성격 구분
```

4번은 첫 구현의 필수 규칙은 아니다. 스킬 수가 늘어나고 선택지가 복잡해졌을 때, 플레이어가 선택의 성격을 빠르게 판단하도록 돕는 가독성 장치다.

## 동전 규칙

첫 버전의 동전 규칙은 단순하게 유지한다.

```text
1. 턴 시작 시 동전 3개를 던진다.
2. 동전은 앞면 또는 뒷면으로 나온다.
3. 스킬은 필요한 앞면/뒷면 개수만큼 동전을 소비한다.
4. 한 턴에 여러 스킬을 사용할 수 있다.
5. 사용한 동전은 즉시 사라진다.
6. 남은 동전은 턴 종료 시 모두 사라진다.
7. 스킬 사용 시 필요한 동전은 자동으로 소비된다.
8. 플레이어가 어떤 개별 동전을 쓸지 직접 고르지 않는다.
```

동전은 성공/실패 판정기가 아니다. 앞면과 뒷면은 이번 턴에 열린 선택 방향을 의미한다.

```text
앞면이 많다
-> 공격, 돌파, 빠른 승리 쪽 선택지가 강해진다.

뒷면이 많다
-> 방어, 수습, 손실 최소화 쪽 선택지가 열린다.

섞여 나온다
-> 균형형 선택지가 중요해진다.
```

## 스킬 비용 규칙

첫 버전의 스킬 비용은 앞면/뒷면 개수만 요구한다.

예시:

```text
앞면 1개
앞면 2개
뒷면 1개
뒷면 2개
앞면 1개 + 뒷면 1개
아무 동전 1개
```

제외:

```text
앞-뒤-앞 같은 순서 패턴 비용
개별 동전 직접 선택
특수 동전
동전 다음 턴 이월
```

## 전사 동전 철학

전사의 앞면과 뒷면은 1:1 화력 밸런스가 아니다. 역할이 다르다.

```text
앞면 = 전사의 주력, 전투를 끝내는 힘
뒷면 = 나쁜 턴을 복구하는 안정 장치
```

전사의 뒷면 스킬은 강해서 선택하는 것이 아니라, 뒷면이 나왔을 때 아무것도 못 하는 불안을 줄이기 위해 선택하게 만드는 것이 목적이다.

현재 확정된 방향:

```text
전사는 앞면으로 전투를 끝내고,
뒷면으로 나쁜 턴을 복구한다.
```

아직 확정하지 않을 것:

```text
뒷면 스킬이 방어만 할지
약한 공격도 포함할지
뒷면 3개 보너스를 줄지
앞면 조건을 보정하는 기능을 줄지
```

이 부분은 실제 스킬 세트와 시뮬레이션 이후 판단한다.

## 선택 유형

동전 시스템에서 선택 유형은 플레이어에게 책임 있는 선택감을 주기 위한 구분이다.

### 안전 선택

낮은 보상이지만 안정적인 선택이다.

```text
아무 동전 1개 -> 방어 4
뒷면 1개 -> 방어 5
앞면 1개 -> 피해 3
```

역할:

```text
운이 나빠도 턴이 완전히 망하지 않게 한다.
```

### 위험 선택

조건이 까다롭지만 성공하면 강한 선택이다.

```text
앞면 2개 -> 피해 12
앞면 3개 -> 피해 20
```

역할:

```text
플레이어가 이번 턴 욕심을 낼지 고민하게 만든다.
```

### 긴급 선택

효율은 낮지만 생존을 보장하거나 큰 피해를 줄이는 선택이다.

```text
모든 동전 소비 -> 방어 10
아무 동전 2개 -> 이번 턴 받는 피해 감소
```

역할:

```text
억울한 죽음을 줄인다.
하지만 남용하면 손해가 있어야 한다.
```

## 기본 턴 흐름

```text
1. 몬스터가 다음 행동과 예상 피해를 보여준다.
2. 플레이어가 파우치를 열어 동전 3개를 확인한다.
3. 현재 남은 동전으로 사용할 수 있는 스킬이 활성화된다.
4. 플레이어가 스킬을 선택한다.
5. 필요한 경우 대상을 선택한다.
6. 스킬 효과가 처리되고 동전이 소비된다.
7. 남은 동전으로 추가 스킬을 사용할 수 있다.
8. 플레이어가 턴 종료를 누른다.
9. 몬스터 행동이 실행된다.
10. 다음 턴으로 넘어간다.
```

## 전투 후 보상 루프

전투 보상은 방금 겪은 문제를 떠올리게 해야 한다.

```text
전투 중:
내 스킬 구성의 약점이 드러난다.

전투 후:
그 약점을 보완할지,
강점을 강화할지,
더 위험한 고점을 노릴지 고른다.
```

보상 선택지는 다음 세 방향을 제공하는 것이 이상적이다.

```text
1. 약점 보완
   예: 뒷면 대응이 부족했으니 안정 스킬을 얻을까?

2. 강점 강화
   예: 앞면 공격이 잘 맞으니 더 공격적으로 갈까?

3. 위험한 고점
   예: 조건은 까다롭지만 터지면 강한 스킬을 배울까?
```

보상은 정답을 주면 안 된다. 보상은 방향 선택을 줘야 한다.

## 시뮬레이터에서 검증할 위험

현재 규칙으로 시뮬레이터를 돌렸을 때 가장 먼저 확인해야 할 문제는 다음과 같다.

```text
1. 선택지가 너무 적어서 고민이 없는가
2. 항상 최적 행동이 뻔한가
3. 뒷면이 나왔을 때 여전히 불쾌한가
4. 전투가 너무 길거나 짧은가
5. 아무것도 못 하는 턴이 발생하는가
6. 항상 동전을 다 쓰는 것이 정답이 되는가
7. 플레이어가 받은 피해를 사전에 예측할 수 있었는가
```

가장 큰 위험:

```text
규칙은 깔끔하지만,
실제 플레이가 "쓸 수 있는 것 다 누르기"가 되는 것
```

이를 막기 위해 스킬은 단순히 효율 순서로 나열되지 않아야 한다. 공격, 방어, 손실 최소화, 미래 위험 감소 사이에 선택 압박이 있어야 한다.

## 1차 스킬 세트 작성 방향

다음 단계는 코드 구현이 아니라 전사 1차 스킬 세트 초안을 만드는 것이다.

권장 규모:

```text
앞면 주력 스킬 3개
뒷면 안정 스킬 3개
혼합 스킬 2개
긴급 스킬 1개
```

각 스킬은 다음 항목으로 정의한다.

```text
SkillName
Cost: 앞/뒤/아무 동전 개수
Effect: 피해 / 방어 / 회복 / 특수
Role: 보완 / 강화 / 고점 / 긴급
Purpose: 어떤 고민을 만들기 위한 스킬인지
```

## 현재 비결정 사항

아래 항목은 아직 결정하지 않는다.

```text
1. 뒷면 스킬의 정확한 강도
2. 방어도 Block 도입 방식
3. 데미지 타입 / 저항
4. 버프 / 디버프 / 상태이상
5. 몬스터의 복잡한 AI 패턴
6. 스킬 보상 생성 알고리즘
7. 전투 종료 후 스킬 보상 UI
```

이 항목들은 전사 1차 스킬 세트와 간단한 시뮬레이션 이후 판단한다.

## 설계 기준 문장

이 문서를 기준으로 하는 현재 전투 설계 문장은 다음과 같다.

```text
전투는 매 턴 동전 3개와 몬스터 intent를 보고,
제한된 자원을 어떤 순서로 쓸지 고민하게 만든다.

전투 후에는 그 과정에서 드러난 스킬 구성의 약점을 보고,
약점 보완 / 강점 강화 / 위험한 고점 중 하나를 선택하게 만든다.
```
